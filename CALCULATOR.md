# Калькулятор доходности — Унистрой.Аренда

Технический и продуктовый референс по `investment_calculator.html`. Цель документа —
дать любой LLM (или человеку) полное понимание того, зачем существует этот
инструмент, как именно он считает деньги и какие в нём есть неочевидные решения
и ограничения, прежде чем что-то в нём менять.

Если после чтения этого файла что-то в коде выглядит странно — сначала проверь
раздел [10. Нюансы, которые выглядят как баги, но не баги](#10-нюансы-которые-выглядят-как-баги-но-не-баги)
и [11. История решений в этом файле](#11-история-решений-в-этом-файле), прежде чем чинить.

## 1. Что это и для кого

Веб-калькулятор для отдела продаж Унистрой.Аренда. Менеджер (или сам клиент)
вводит сумму, которую готов вложить, — калькулятор показывает, что выгоднее:

- купить квартиру (одну или несколько) у Унистроя и сдавать её через УК одним из
  трёх продуктов (LTR / Гарантированная / STR) в пределах операционной
  применимости конкретной комнатности, или
- положить те же деньги на банковский вклад.

Результат — не просто цифра, а сравнение по годам (таблица + график), с честным
учётом ипотеки (если бюджета на квартиру не хватает), налогового/комиссионного
устройства каждого продукта аренды и роста стоимости жилья.

Инструмент **не является публичной офертой** (дисклеймер внизу страницы) и не
привязан к конкретному юрлицу/сделке — расчёт оценочный.

Бизнес-контекст продуктов (комиссии УК) — см. корневой `CLAUDE.md` проекта,
раздел 2. Здесь только то, как это реализовано в коде.

## 2. Где живёт и как разворачивается

- Единый файл `investment_calculator.html` — HTML + CSS + JS в одном файле,
  **без сборки, без npm, без бандлера**. Открывается напрямую в браузере или
  раздаётся статикой.
- Хостится на GitHub Pages: `https://anton-shrmt.github.io/invest-calc/investment_calculator.html`.
- Деплой = обычный `git push` в этот репозиторий (`website/calculator/` —
  **отдельный git-репозиторий**, не тот же, что корень проекта «Унистрой
  Аренда» — см. раздел 12).
- Данные о квартирах подгружаются отдельным сгенерированным JS-файлом
  (`scripts/output/calculator_projects_data.js`) через `<script>`, который
  калькулятор сам дописывает в `<head>`/тело документа при загрузке.

## 3. Архитектура файла

Всё внутри `investment_calculator.html`:

1. `<style>` — CSS. Дизайн-токены в `:root` (свои, **не** совпадают 1:1 с
   брендбуком Унистроя из корневого `CLAUDE.md` — у калькулятора отдельная,
   более сдержанная палитра `--dark #364B4B`, `--accent #A5FF00` и т.д., но
   структурно похожая).
2. Разметка `<body>` — статичные карточки-контейнеры, все текстовые значения
   внутри них заполняются JS (`textContent`/`innerHTML`), исходно пустые или с
   плейсхолдером `—`.
3. `<script>` (первый, короткий) — динамически дописывает `<script src="scripts/output/calculator_projects_data.js?v=...">`,
   где `v` — таймстамп, округлённый до 12-часового окна (=не даёт браузеру
   отдать вчерашний кэш; сами данные и так обновляются раз в 12 часов).
4. `<script>` (второй, основной, ~1600 строк) — вся логика:
   - константы данных (цены ЖК, ставки аренды, экономика STR),
   - утилиты форматирования,
   - объект `Calc` (единственный модуль всего приложения, без классов/фреймворков),
   - `Calc.init()` в самом конце — точка входа.

Никаких модулей/импортов в браузере — всё в одной глобальной области видимости
одного `<script>`.

## 4. Источники данных

### 4.1 Квартиры/цены — `PROJECTS`

```js
const PROJECTS = window.CALCULATOR_PROJECTS || [ /* встроенный fallback-снимок */ ];
```

`window.CALCULATOR_PROJECTS` подгружается из `scripts/output/calculator_projects_data.js`
(сгенерирован скриптом, см. 4.3). Если файл почему-то не подгрузился —
используется встроенный в HTML массив-снимок (на момент написания — данные на
2026-07-13), чтобы страница не осталась пустой.

Структура одного элемента `PROJECTS`:

```js
{
  slug: 'aqua', city: 'kzn', label: 'Аквамарин',
  rooms: [
    {
      label: '1',                    // 'Студия' | '1' | '2' | '3' | '4' | '5'
      count: 11,                     // сколько лотов этой комнатности сейчас в продаже
      priceMin, priceMax, priceAvg,  // по всем лотам этой комнатности
      priceP25, areaP25,             // цена/площадь лота в 25-м перцентиле (метод ближайшего ранга —
      priceP50, areaP50,             // ЦЕНА И ПЛОЩАДЬ ВСЕГДА ОТ ОДНОГО И ТОГО ЖЕ РЕАЛЬНОГО ЛОТА,
      priceP75, areaP75,             // не среднее по одному + среднее по другому)
      areaMin, areaMax,
    },
  ],
}
```

**Расчёт использует `priceP50` (медианный реальный лот) как «типичную цену»**
(`priceForTier()`: `room.priceP50 ?? room.priceAvg` — `priceAvg` только
если P50 почему-то отсутствует в данных). Площадь для того же лота —
`areaForTier()`: `room.areaP50 ?? (areaMin+areaMax)/2`. `priceMin`/`priceMax` —
только для отображения диапазона, в формулах не участвуют.

### 4.2 Ставки аренды — `RENT_RATES` + `RENT_YIELD`

Два источника, в порядке приоритета:

1. **`RENT_RATES[projectSlug][roomLabel]`** — реальный прайс-лист УК
   (`{ daily, monthly }`, ₽), источник — `цены_для_калк.xlsx`. Покрывает
   только Студия/1к/2к и не все ЖК.
2. **`RENT_YIELD[roomLabel]`** — фолбэк, если для ЖК+комнатности нет строки в
   `RENT_RATES`. Это % от цены квартиры в год:

   | Комнатность | LTR (`ltr`) | STR (`daily`) |
   |---|---|---|
   | Студия | 7.0% | 11.5% |
   | 1 | 6.0% | 9.8% |
   | 2 | 4.6% | 7.6% |
   | 3 | 3.8% | 6.2% |
   | 4 | 3.2% | 5.2% |
   | 5 | 2.8% | 4.5% |

   Если и это не нашлось (не должно случаться, но код это допускает) —
   `RENT_YIELD['1']` как последний фолбэк.

Когда используется фолбэк — `getRentFigures()` возвращает `isEstimated: true`,
и UI показывает пометку «доход оценочный» / «Доход рассчитан оценочно по
доходности комнатности: подтвердите ставку у УК».

### 4.3 Экономика STR (посуточной аренды)

Не просто «ставка × дни», а полная модель операционной прибыли:

```js
STR_RENTED_DAYS      = 265;    // из 365 — простой между заездами/сезонность
STR_AGGREGATOR_FEE    = 0.15;  // комиссия агрегатора (Авито/Островок и т.п.) от валовой выручки
STR_CLEANING_PER_DAY  = 500;   // ₽/сутки аренды — уборка между заездами
STR_OTHER_ANNUAL_COST = 126400;// ₽/год фикс — коммуналка, расходники, страховка, фотосъёмка, продвижение
STR_OWNER_SHARE       = 0.72;  // после операционных расходов собственник забирает 72% (28% — комиссия УК)
```

```
grossRevenue     = dailyRate × 265
aggregatorFee    = grossRevenue × 0.15
cleaningCost     = 500 × 265
operatingProfit  = grossRevenue − aggregatorFee − cleaningCost − 126400
dailyAnnualIncome = max(0, operatingProfit × 0.72)
```

`STR_OWNER_SHARE = 0.72` **соответствует** заявленной в бизнес-модели комиссии
STR «28% от чистой прибыли» (см. корневой `CLAUDE.md`, раздел 2) — это одно и
то же число, не рассинхронизировано.

Калибровка (см. комментарий в коде): базовый сценарий 5 000 ₽/сутки, 265 дней
→ ≈624 500 ₽ владельцу в год. Это же число проверяется в
`tests/calculator_regression.mjs`.

### 4.4 Пайплайн обновления цен

`scripts/fetch_unistroy_prices.mjs` (Node 18+, без зависимостей):

1. Тянет **все** лоты со всех городов с публичного `unistroy.ru/api/flats/`
   (постранично, `limit=1000`), с ретраями (до 5 попыток, экспоненциальная
   задержка, не ретраит 4xx кроме 429).
2. Требует правдоподобный `User-Agent` — без него unistroy.ru отдаёт 403
   (простая защита от ботов, не капча).
3. Группирует лоты по `(city_code, project.slug, комнатность)`, считает
   `priceMin/Max/Avg` и P25/P50/P75 (метод ближайшего ранга — см. 4.1).
4. Фильтрует до `CALCULATOR_PROJECT_SLUGS` — **явный allowlist** ЖК, для
   которых утверждены ставки аренды (`art16`, `yes_gorki`, `upoint` и все
   остальные ЖК unistroy.ru **намеренно** исключены).
5. Исключает квартиры от 3 комнат и больше везде — не целевой продукт
   инвестора.
6. Пишет три файла в `scripts/output/`:
   - `projects_data.js` — полный массив по всем ЖК (не только allowlist), для
     сверки;
   - `calculator_projects_data.js` — то, что реально грузит калькулятор
     (`window.CALCULATOR_PROJECTS` + `window.CALCULATOR_PROJECTS_META`);
   - `summary.json` — то же плюс сырые тоталы по городам.

**Автообновление**: `.github/workflows/update-prices.yml`, cron `0 */12 * * *`
(каждые 12 часов), коммитит `scripts/output/`, если что-то изменилось. Ручной
запуск — вкладка Actions → Update flat prices → Run workflow.

**Важно для агента, запускающего скрипт через Bash-тул Claude Code**: обычный
sandboxed Bash не имеет выхода в интернет — нужен `dangerouslyDisableSandbox: true`.

**Нестабильность `city_code` у unistroy.ru** (реальный инцидент, 2026-07-22):
API один раз без предупреждения стал отдавать полное название города вместо
короткого кода для части городов (`mahachkala` вместо `mhc`, `perm` вместо
`per`). Это тихо ломало `PROJECTS.filter(p => p.city === 'mhc')` — калькулятор
падал при выборе города. Нормализуется в скрипте:

```js
const CITY_CODE_ALIASES = { mahachkala: 'mhc', perm: 'per' };
```

При появлении новых алиасов — дополнить этот список. Плюс на стороне
клиента добавлены защитные проверки каскада город→ЖК→комнатность против
реально живых данных (см. раздел 9.3 и пункт 6 раздела 10).

## 5. Финансовая модель: доход по каждой стратегии

Все формулы — на **одну квартиру за год**, затем масштабируются на количество
купленных квартир (`units`, см. раздел 6) и растут год к году на `rentGrowth`
(% в год, задаёт пользователь).

```
g(year) = (1 + rentGrowth/100) ^ (year − 1)     // рост арендной ставки к году `year`, year=1 → g=1
```

### 5.1 Долгосрочная аренда (LTR)

```
yr(year) = monthlyRent × 11 × 0.88 × g(year)
```
11 месяцев (условно 1 месяц простой между жильцами при переезде), минус 12%
комиссии УК (`0.88 = 1 − 0.12`).

### 5.2 Гарантированная аренда

```
yr(year) = monthlyRent × 12 × 0.75 × g(year)
```
Полные 12 месяцев (простой не важен — собственнику платят гарантированно),
минус 25% комиссии УК (`0.75 = 1 − 0.25`).

### 5.3 Посуточная аренда (STR)

```
yr(year) = dailyAnnualIncome × g(year)
```
`dailyAnnualIncome` уже посчитан один раз в `getRentFigures()` (см. 4.3) —
чистый доход собственника ЗА ГОД, комиссия УК/агрегатора/расходы уже вычтены.

### 5.4 Операционная применимость моделей аренды

Калькулятор не рекомендует математически выгодный, но неисполняемый сценарий.
Матрица `RENTAL_PRODUCT_MATRIX` определяет доступные продукты по комнатности:
LTR и гарантированная аренда разрешены для всех комнатностей, а STR — только
для студий и 1-комнатных квартир. Поэтому 2-комнатная квартира никогда не
получит рекомендацию «посуточная аренда», даже если гипотетическая формула
дала бы более высокий доход. Точечное исключение по ЖК можно добавить в
`byProject` только после подтверждения операционной командой.

В основной таблице и на графике недоступная модель не показывается, а в блоке
«Аренда» выводится объяснение, почему посуточный сценарий исключён.

### 5.5 Рост стоимости жилья по городу

Поле `appreciation` не редактируется вручную: оно синхронизируется с выбранным
городом. Согласованные ставки: Казань — 13%, Махачкала — 18%, Екатеринбург —
13%, Санкт-Петербург — 11%, Пермь — 12%, Тольятти — 14%. Если в будущем в
калькулятор добавят город без явной ставки, текущее значение сохраняется до
согласования его допущения.

### 5.6 Банковский вклад

Считается **от всей введённой суммы `investAmount`**, а не от стоимости
квартиры — это намеренно, чтобы сравнение было честным «вложить всё в
недвижимость + остаток кэшем» против «вложить всё во вклад»:

```
С ежемесячной капитализацией: investAmount × (1 + depositRate/100/12) ^ (year × 12)
С годовой капитализацией:     investAmount × (1 + depositRate/100) ^ year
```

## 6. Покупка одной или нескольких квартир

### 6.1 Политика распределения — `allocateUnits(budget, price, targetUnits)`

Общая функция, которую используют **и** основной расчёт (`compute()`), **и**
карточки-подсказки (`_computeHint()` → `evalCombo()`). Раньше это была
задублированная логика в двух местах — унифицировано.

Правило (бизнес-решение, не подлежит «оптимизации» без обсуждения):

> Максимум целых квартир **за наличные**, остаток бюджета — **в ипотеку**,
> но **не больше 2 квартир в ипотеку одновременно**, и только если на каждую
> хватает минимум **30% первоначального взноса** от её стоимости.

```js
MIN_DOWN_PAYMENT_SHARE = 0.30;
MAX_MORTGAGED_UNITS    = 2;

nFullMax           = floor(budget / price)
cashLeftMax         = budget − nFullMax × price
mortgageCapacityMax = floor(cashLeftMax / (price × 0.30))
mortgagedUnitsMax   = min(mortgageCapacityMax, 2)
naturalUnits        = nFullMax + mortgagedUnitsMax        // «естественный» максимум под бюджет
```

Весь остаток кэша сверх нужных 30% идёт в **увеличенный первоначальный
взнос** по ипотеке (`principal = mortgagedUnits × price − cashLeft`) — так
меньше тело кредита и переплата, а не оставляется «просто лежать».

### 6.2 Поле «Количество квартир» (`unitsOverride`)

Пользователь может явно задать количество квартир вместо автомакс —
`d.unitsOverride` (0 = авто, положительное число = ручное значение,
1–20 после санитайза).

- **Если `targetUnits ≤ naturalUnits`** (бюджета хватает): используется ровно
  запрошенное количество, разбитое той же политикой (кэш максимум, остаток —
  в ипотеку до потолка 2). Доказано (см. комментарии в коде), что это ВСЕГДА
  укладывается в бюджет без пересчёта заново — потому что `naturalUnits` уже
  посчитан по той же policy.
- **Если `targetUnits > naturalUnits`** (бюджета не хватает): **не блокируем
  ввод**. Показываем `naturalUnits` (то, что реально доступно) как основной
  расчёт, плюс `shortfall` — сколько ₽ не хватает, чтобы дотянуть именно до
  `targetUnits`. Логика shortfall: сначала «дозаполняем» оставшиеся слоты
  ипотеки (дешевле — только 30% взноса), а сверх лимита 2 квартир — уже
  только за полную стоимость наличными.
- Поле в UI (`#units-input`) всегда показывает **запрошенное** число (эхо
  ввода пользователя), даже если оно недостижимо — отдельная строка
  `#units-shortfall` объясняет расхождение. Реальное используемое количество
  видно в «Типичная цена за N квартир» и в баннере/таблице ниже.
- Смена ЖК, комнатности или региона **сбрасывает** `unitsOverride` в 0 (авто)
  — старое количество квартир не имеет смысла для другой цены/объекта.
- Ручное изменение количества **помечает** объект как выбранный вручную
  (`objectManualOverride = true`, см. раздел 8) — точно так же, как ручной
  выбор ЖК или комнатности.

### 6.3 Ипотека — аннуитетный платёж

```
mr = mortgageRate / 12 / 100     // месячная ставка
n  = mortgageYears × 12          // срок в месяцах

monthlyPayment = principal × mr × (1+mr)^n / ((1+mr)^n − 1)
```
(запасной путь `principal / n`, если `mr === 0`, — на практике недостижим,
т.к. `mortgageRate` санитайзится в диапазон [1, 40]%, но оставлен для
устойчивости формулы).

Остаток долга на конец года `y` (месяцев `m = min(y×12, n)`):

```
growth  = (1+mr)^m
balance = max(0, principal × growth − monthlyPayment × (growth−1) / mr)
```
= 0, если `m ≥ n` (кредит уже выплачен на этом горизонте).

**Ипотечный годовой платёж вычитается из арендного дохода каждый год, пока
`year ≤ mortgageYears`** — после погашения кредита денежный поток становится
полностью положительным (только рента, без вычета платежа).

## 7. Итоговый капитал, ROI, победитель, окупаемость

Для каждой стратегии аренды (LTR / Гарант / STR) и для вклада считается
массив по годам `0..H` (`H` = горизонт планирования):

```
totalCost   = unitPrice × units
propArr[y]  = totalCost × (1 + appreciation/100) ^ y          // стоимость недвижимости на год y
equityArr[y] = propArr[y] − mortgageBalanceArr[y]              // капитал за вычетом остатка долга
cashArr[y]  = cashRemainder + Σ(net_i, i=1..y)                 // накопленный денежный поток
                где net_i = (rentPerUnit_i × units) − (ипотечный платёж, если год ≤ mortgageYears)
wealthArr[y] = equityArr[y] + cashArr[y]                       // ИТОГОВЫЙ КАПИТАЛ на год y

ROI = (wealthArr[H] − investAmount) / investAmount × 100
```

**Победитель** — модель (из LTR/Гарант/STR/Вклад) с максимальным
`wealthArr[H]`. Если объект не проходит по бюджету (`isAffordable === false`,
т.е. нельзя купить даже дробную часть одной квартиры — меньше 30% от
стоимости) — в сравнении участвует только вклад.

**Срок окупаемости** (`paybackYears`, простая недисконтированная метрика):
сколько лет накопленная (растущая на `rentGrowth`) годовая рента идёт до
полной стоимости покупки (`totalCost`). Считается только для
победителя-аренды, максимум 60 лет, иначе «свыше 60 лет».

## 8. Два блока-подсказки: «Лучший вариант» и «Ваш выбор»

Отдельная, более простая модель поиска — **не переиспользует** массивы по
годам из раздела 7, а считает одно число (`wealth` на конец горизонта) для
множества кандидатов сразу, чтобы можно было их сравнивать и сортировать.
Живёт в `Calc._computeHint()` / `Calc._renderHint()`.

### 8.1 Как ищется «лучший вариант»

`_computeHint()` перебирает **все ЖК × комнатности внутри ТЕКУЩЕГО выбранного
города** (`d.cityCode` — в другие города поиск не заходит, смена региона это
осознанное действие пользователя, а не то, что подсказка должна предлагать
сама), пропуская комнатности с `count < 3` активных лотов (не показательны как
«типовая» рекомендация). Плюс всегда в списке кандидатов — банковский вклад.

Для каждого кандидата — `evalCombo(project, room, budget, targetUnits)`:
методологически то же самое, что и в `compute()` (`allocateUnits`, аннуитет,
рост стоимости), но:

- **группа купленных-за-нал и группа купленных-в-ипотеку квартир может
  каждая выбрать СВОЮ лучшую модель аренды** (LTR/Гарант/STR) независимо друг
  от друга — в отличие от `compute()`, где ОДНА модель применяется ко всей
  покупке целиком. Это осознанная, более «оптимистичная» гранулярность
  именно для карточки-рекомендации — из-за этого сумма чисел в карточке и в
  основном расчёте для ОДНОГО и ТОГО ЖЕ объекта может чуть отличаться, если
  покупка смешанная (часть на нал + часть в ипотеку). Это не баг.

Все кандидаты сортируются по `wealth` по убыванию — `strategies[0]` = `best`.

### 8.2 Как считается «Ваш выбор»

Отдельно, **всегда пересчитывается напрямую** (не берётся из списка выше, даже
если та же связка ЖК+комнатность там уже встречалась) —
`evalCombo(selProject, selRoom, invest, d.unitsOverride)` — потому что нужно
учесть именно пользовательский `unitsOverride`, а не авто-количество.

### 8.3 Рендер — `_renderHint()`

- **Верхняя карточка** («Лучший вариант под ваш бюджет») — **всегда** `best`.
  Никогда не подменяется тем, что выбрано в «Объект недвижимости» ниже.
- **Нижняя карточка** («Ваш выбор») показывается **только если**:
  `objectManualOverride === true` **и** `selected` существует **и**
  (`selected.key !== best.key` **или** `selected.units !== best.units`) —
  то есть только когда есть что реально сравнивать. Совпадение по ЖК/комнате,
  но разное количество квартир — тоже считается «другим вариантом» (ключ
  `key` не включает количество, поэтому сравнение количества добавлено
  отдельным условием — важно не забыть при рефакторинге).

## 9. Состояние приложения, синхронизация, персистентность

### 9.1 `Calc.data`

Единственный источник правды. Поля: `investAmount, cityCode, projectSlug,
roomLabel, objectManualOverride, unitsOverride, rentGrowth, appreciation,
horizon, depositRate, depositMonthly, mortgageRate, mortgageYears,
managerName, managerPhone`.

### 9.2 `objectManualOverride` — сердце UX-логики «Объект недвижимости»

- **`false` (авто, по умолчанию)** — `projectSlug`/`roomLabel` **всегда**
  следуют за `best` из раздела 8.1. Как только меняется бюджет, город или
  любое допущение (рост аренды/стоимости, ставка вклада и т.д.) — если
  найденный `best` отличается от текущего выбора, `_syncSelectionToBest()`
  тихо переключает селекторы «ЖК»/«Комнатность» на него и заново их
  перерисовывает (`_populateProjectSelect`/`_populateRoomSelect`).
- **`true` (ручной режим)** — включается, как только пользователь сам
  трогает селект ЖК, комнатности или поле «Количество квартир», ЛИБО
  приходит по ссылке с параметром `proj`/`room`/`qty` в URL.
  `_syncSelectionToBest()` в этом состоянии — no-op.
- **Сброс обратно в `false`**: смена региона (город — это «начать заново»),
  либо кнопка «Вернуть значения по умолчанию».

`_syncSelectionToBest()` вызывается: один раз синхронно в конце `init()`
(чтобы при самой первой отрисовке страницы уже стоял лучший вариант, без
мигания), и первой строкой внутри задебаунсенного `_recompute()` — то есть
после **любого** изменения любого поля, если оно не привело к ручному
переключению.

### 9.3 localStorage + URL-параметры

- `localStorage['calc_v5']` — весь `Calc.data` целиком как JSON,
  перезаписывается на каждое изменение (`_saveLS()`).
- URL-параметры (`_urlParamMap`) **приоритетнее** localStorage при загрузке —
  чтобы получатель ссылки-расчёта от менеджера видел именно присланный
  сценарий, а не свою предыдущую сессию:

  | Параметр | Поле |
  |---|---|
  | `amt` | investAmount |
  | `city` | cityCode |
  | `proj` | projectSlug |
  | `room` | roomLabel |
  | `qty` | unitsOverride |
  | `rg` | rentGrowth |
  | `h` | horizon |
  | `dr` | depositRate |
  | `dm` | depositMonthly (`'1'`/`'0'`) |
  | `mr` | mortgageRate |
  | `my` | mortgageYears |
  | `mn` | managerName |
  | `mp` | managerPhone |

  `buildShareUrl()` собирает эту же ссылку — используется кнопками
  «Скопировать ссылку», QR-код, и текстом сообщения WhatsApp.

- `_loadLS()` порядок: localStorage → URL-параметры поверх → `_sanitizeData()`
  (клампит все числа в разумные границы) → **валидация каскада** город → ЖК →
  комнатность против **реально живых** `PROJECTS` (если сохранённый город
  вдруг не имеет ни одного активного ЖК — переключаемся на первый доступный;
  так же для ЖК/комнатности). Это защитный код, добавленный после инцидента с
  нестабильным `city_code` (раздел 4.4) — **не убирать**, даже если кажется
  избыточным.

### 9.4 Кнопка «Вернуть значения по умолчанию»

Полностью пересобирает `Calc.data` с нуля (кроме `managerName`/`managerPhone`
— это профиль продажника, не параметр расчёта) и делает
`window.location.href = window.location.pathname` — **полный переход без
query-параметров**, а не просто `location.reload()`, иначе URL-параметры от
предыдущей ссылки «Поделиться» тут же снова бы себя применили.

## 10. Нюансы, которые выглядят как баги, но не баги

1. **Политика «сначала нал» не глобально оптимальна по числу квартир.**
   Для некоторых бюджетов теоретически можно было бы купить БОЛЬШЕ квартир,
   если сразу максимально использовать ипотечный рычаг (30% взнос дешевле
   100% нала), а не тратить нал на целые квартиры первым делом. Текущий
   алгоритм этого не ищет — это осознанный риск-менеджмент (меньше долговой
   нагрузки), а не недоработка.

2. **«Лучший вариант» ищет только в пределах выбранного города.** Никогда не
   предложит более доходный ЖК в другом регионе — смена региона это выбор
   пользователя.

3. **Карточки-подсказки и основной расчёт могут чуть разойтись в цифрах**
   для одного и того же смешанного (нал + ипотека) объекта — потому что
   `evalCombo` позволяет разным группам квартир выбрать разную модель аренды,
   а `compute()` — нет (см. 8.1).

4. **Поле «Количество квартир» показывает запрошенное число, а не
   фактическое**, если бюджета не хватает — фактическое видно ниже по
   странице (в «Типичная цена», баннере, таблице). Предупреждение о нехватке
   — отдельной строкой под самим полем.

5. **RENT_YIELD — это фолбэк, не основной источник.** Если ставка вдруг
   выглядит «слишком круглой/усреднённой» — проверить `isEstimated` в выдаче
   `getRentFigures()`; если `true`, значит для этого ЖК+комнатности нет
   строки в `RENT_RATES`, и это ожидаемо.

6. **`city_code` от unistroy.ru нестабилен** (раздел 4.4). Если после
   автообновления цен вдруг «пропал» город из списка — сначала проверить,
   не начал ли API отдавать город под новым алиасом, прежде чем считать это
   багом калькулятора.

7. **Вклад считается от полной `investAmount`**, а не от стоимости квартиры —
   намеренно, для честного сравнения «всё в недвижимость» vs «всё во вклад».

8. **Модель «Сдаю сам» была в коде и удалена по прямому запросу
   пользователя** (2026-07-22). Если увидите упоминания в старых
   коммитах/бэкапах — не восстанавливать без явного запроса.

9. **Дисклеймер «Разница незначительна. Учитывайте риски.»** тоже удалён по
   явному запросу — раньше показывался в баннере, когда топ-2 стратегии были
   близки по итогу (<5% разницы). Сейчас баннер всегда просто показывает
   победителя без смягчающей оговорки.

10. **Задвоение ипотеки в карточках-подсказках** — исторический баг (уже
    исправлен, коммит `999392c`): при покупке 2 квартир в ипотеку остаток
    долга и годовой платёж считались один раз «на группу», а затем результат
    ещё раз умножался на количество квартир. Если после рефакторинга
    `groupWealth`/`cumRentFn`/`allocateUnits` карточки вдруг снова начнут
    показывать неправдоподобно отрицательный итог при 2 ипотечных квартирах —
    проверить это в первую очередь.

## 11. История решений в этом файле

Коротко, зачем текущая архитектура такая, какая есть (полные объяснения — в
истории git-коммитов этого репозитория):

- **Автоподбор «лучшего варианта» вместо жёстко закодированного ЖК по
  умолчанию.** Раньше дефолтный сценарий (Аквамарин, 1-комн., бюджет 8М)
  жёстко указывал на конкретный ЖК — из-за бага с задвоением ипотеки (см.
  п.10) это давало пугающий отрицательный итог на самой первой загрузке
  страницы. Переход на автоподбор лучшего варианта решил и косметическую, и
  расчётную проблему одновременно.
- **Разделение «Лучший вариант» / «Ваш выбор» на два блока.** Раньше была
  ОДНА карточка, которая тайком подменяла выбранный пользователем объект на
  объективно лучший — заголовок менялся («Прогноз для выбранного объекта» ↔
  «Лучшая альтернатива»), но бейдж/цифры относились уже к другому ЖК.
  Пользователь терял из виду то, что сам выбрал. Сейчас — два независимых,
  всегда честно подписанных блока.
- **`allocateUnits()` как общая функция.** Раньше и `compute()`, и
  `evalCombo()` (внутри `_computeHint`) содержали дублированную и слегка
  разошедшуюся логику распределения нал/ипотека — вынесено в одну функцию
  верхнего уровня, используется обоими.
- **Защитные проверки city→ЖК→комнатность.** Добавлены после того, как смена
  формата `city_code` от unistroy.ru (см. 4.4) один раз реально сломала
  страницу для двух городов в проде.

## 13. Тесты

```bash
node tests/calculator_regression.mjs
```

Загружает `calculator_projects_data.js` и вытаскивает **реальный** inline
`<script>` из `investment_calculator.html` в `vm`-контекст (не копия логики —
буквально тот же код, что исполняется в браузере), экспортирует `Calc` и
гоняет assertions:

- посуточный доход по базовому сценарию из ТЗ (≈624 500 ₽/год);
- границы санитайза (`investAmount`, `mortgageRate`, `mortgageYears`);
- типичная цена = `priceP50`;
- минимальный взнос = ровно 30% от цены квартиры (граница платёжеспособности);
- потолок в 2 квартиры в ипотеку одновременно, даже если бюджета хватает на
  больше (только за нал сверх этого);
- `unitsOverride`: меньше натурального максимума → используется ровно
  запрошенное, остаток бюджета не теряется; больше максимума → не
  блокируется, но `targetAffordable=false` и считается `unitsShortfall`.

**Запускать после любого изменения формул/логики распределения** — тест
дешёвый (доли секунды, без сети и без браузера).
